## 二、根因分析

### 2.1 SQLite 单文件数据库的天花板

SQLite 适合嵌入式场景，但本平台的使用模式已经超出它的舒适区：

| 场景 | SQLite 表现 | 问题 |
|---|---|---|
| 10万行告警 + 并发读写 | 写锁独占，读也可能被阻塞 | 导入时其他请求全部等待 |
| 复杂聚合（热度/Badge/评分） | 无并行查询引擎 | 单线程逐行扫描 |
| 频繁的小事务（标签/事件/追踪） | WAL 模式有帮助，但 checkpoint 成本高 | 阶段 23 专门加了定时 checkpoint |
| 全文搜索 `LIKE '%keyword%'` | 全表扫描 | 10万行每次搜索都要遍历 |

SQLite 引发的连锁问题：
- 因为查询慢 → 加了进程内 Python 缓存
- 因为有缓存 → 数据变更后必须手动失效缓存
- 因为缓存失效遗漏 → 标签/事件不显示
- 因为缓存失效太频繁 → 加了快照预计算表
- 因为快照和实时数据不一致 → 加了实时覆盖层
- 覆盖层又引入新的一致性问题 → 继续补丁...

### 2.2 "快照 + 实时覆盖"混合模型的死结

当前架构（阶段 70-77 确认）：

```
快照表（预计算的静态数据）
    ↓ SQL 查询、过滤、排序、分页
    ↓
快照行（可能是旧数据）
    ↓ Python 实时覆盖（事件/标签/追踪/Badge/评分）
    ↓
最终响应（和 SQL 层的过滤/排序/总数可能不一致）
```

**这是两种策略的最差组合：**
- 快照的好处（SQL 快）被实时覆盖（Python 慢）抵消
- 实时查询的好处（数据准）被快照过滤（旧数据）破坏
- 两层之间的一致性无法保证，是不可修复的架构缺陷

### 2.3 God Object / God File

`alerts.py` 单文件 2426 行、63 个函数，承担了：
1. HTTP 路由处理
2. 缓存管理（3 套缓存）
3. SQL 查询构建（实时 + 快照两套 WHERE builder）
4. 数据装饰（热度/Badge/评分/排序信号）
5. 实时关系覆盖
6. 排序逻辑
7. 过滤逻辑（日期/关键词/目标类型/Badge）
8. 筛选选项聚合
9. CSV 导出

任何一处修改都可能影响其他功能。

### 2.4 前端单组件过大

`Workbench.vue` (1758行) 和 `Settings.vue` (2443行) 没有任何组件拆分，所有逻辑都在一个文件里。Vue 3 Composition API 的 `setup` 函数可能超过 1000 行，难以维护。

## 三、方案对比

### 方案 A：在现有代码上继续打补丁

**工作量：** 每个问题 1-2 天
**风险：** 每修一个问题可能引入新问题
**适用：** 只需要再撑 1-2 个月

**不推荐理由：** 过去 82 个阶段已经证明这条路走不通。代码复杂度已超过单人可维护的上限。

### 方案 B：渐进式重构（推荐）

**核心思路：** 不换语言、不换框架，但重新组织代码结构和数据流。

**工作量：** 2-3 周集中重构
**风险：** 中等，可以逐步切换
**适用：** 团队熟悉 Python/Vue，希望保留现有投入

#### 重构路线图

**第一步：砍掉快照表，回归实时查询（3天）**

快照 + 实时覆盖是当前 80% 问题的根源。砍掉后：
- 删除 `snapshot_builder.py`（1246 行）
- 删除 `alerts.py` 中所有快照相关代码（~800 行）
- 删除 `snapshot.py` 模型
- 所有增量 patch 逻辑不再需要
- 事件/标签/追踪变更不再需要同步快照

**第二步：换 PostgreSQL（2天）**

SQLite → PostgreSQL 解决并发和性能根因：
- 真正的并发读写（MVCC），导入不阻塞查询
- 并行查询执行器，10万行聚合不再是瓶颈
- `GIN` 索引支持全文搜索，替代 `LIKE '%x%'`
- 物化视图（`MATERIALIZED VIEW`）原生支持预计算，替代手工快照
- `LISTEN/NOTIFY` 可替代进程内缓存失效

迁移成本低：SQLAlchemy ORM 已经抽象了 SQL 差异，主要改动：
- `db.py` 连接字符串改为 `postgresql://`
- 少量 SQLite 专属语法（`PRAGMA`、`datetime('now')`）替换
- 部署多一步 PostgreSQL 安装

**如果必须保持"单机离线、零依赖部署"的定位，可以不换 PostgreSQL，改用下面的方法：**

**第二步（替代）：SQLite 读写分离 + 物化视图思路（2天）**

不换数据库，但改变使用方式：
- 写操作（导入/标签/事件）走主连接，同步提交
- 读操作（候选查询）走只读连接（`?mode=ro`），不被写锁阻塞
- 用 SQLite 临时表或 `CREATE TABLE AS SELECT` 模拟物化视图
- 导入完成后一次性刷新物化表，查询直接读物化表

**第三步：拆分 alerts.py（2天）**

```
backend/api/alerts.py (2426行)
    ↓ 拆分为
backend/api/
    alerts_list.py          # GET /api/alerts 原始告警列表
    candidates.py           # GET /api/alert-candidates 候选查询
    alerts_export.py        # CSV 导出
backend/services/
    candidate_decorator.py  # 热度/Badge/评分/排序信号计算
    candidate_filter.py     # WHERE 构建 + 筛选选项聚合
    relation_resolver.py    # 事件/标签/追踪关系解析
    candidate_cache.py      # 缓存管理（如果还需要）
```

每个文件 200-400 行，职责单一，修改一个不影响其他。

**第四步：前端组件拆分（2天）**

```
Workbench.vue (1758行)
    ↓ 拆分为
views/Workbench.vue              # 页面壳，组合子组件
components/workbench/
    FilterBar.vue                # 顶部筛选栏
    CandidateTable.vue           # 主表格
    ColumnHeaderFilter.vue       # 表头筛选下拉
    CreateEventDialog.vue        # 创建事件对话框
    SortButton.vue               # 排序按钮
composables/
    useCandidateData.js          # 数据加载/筛选/排序/分页逻辑
    useColumnFilters.js          # 列筛选状态管理
```

**第五步：简化缓存策略（1天）**

当前有 3 套缓存（`_candidate_cache`、`_full_cache`、`sorted_views`），全部是手工管理，失效逻辑散落在 4 个文件中。

简化为：
- 如果用了 PostgreSQL：完全不需要进程内缓存，数据库够快
- 如果继续用 SQLite：只保留一个 LRU 缓存 + 一个全局版本号，任何写操作递增版本号，读操作发现版本号变化就重新查询

### 方案 C：换语言重写

**候选语言：**

| 语言 | 优势 | 劣势 |
|---|---|---|
| **Go** | 编译为单二进制、并发模型好、SQLite 内嵌性能高 | 团队学习成本、Web 框架生态弱于 Python |
| **Rust** | 极致性能、内存安全 | 学习曲线陡峭、开发速度慢 |
| **Node.js (TypeScript)** | 前后端统一语言、异步 I/O 好 | 计算密集型任务（评分/Badge）不如 Python |
| **C# (.NET)** | 性能好、类型安全、单文件部署 | 生态偏企业级、团队可能不熟 |

**工作量：** 4-8 周
**风险：** 高，需要重新实现所有业务逻辑
**适用：** 有长期产品计划、团队愿意投入

**不推荐理由：** 当前问题不是 Python/FastAPI 的语言性能瓶颈，而是架构设计问题。换语言不解决架构问题，只会在新语言中重新制造同样的混乱。

## 四、推荐方案：B（渐进式重构）

### 理由

1. **问题在架构不在语言。** Python + FastAPI 处理 10 万行数据 + 10 个并发用户绰绰有余。瓶颈在 SQLite 写锁、快照一致性、God File，这些换语言也解决不了。

2. **保护已有投入。** 82 个阶段积累的业务逻辑（评分规则、Badge 计算、IOC 提取、去重策略）是核心资产，重写意味着全部重新测试。

3. **可以逐步交付。** 每完成一步都可以上线验证，不需要一次性大爆炸切换。

4. **解决根因而非症状。** 砍掉快照表消除一致性问题的根源；换数据库（或改用法）消除并发问题的根源；拆分文件消除可维护性问题的根源。

### 预期效果

| 指标 | 当前 | 重构后 |
|---|---|---|
| 候选页加载 | 首次 ~110s（快照构建），后续依赖缓存 | ≤ 3s（PostgreSQL）或 ≤ 5s（SQLite 物化表） |
| 上传 10 万行 | 经常卡死其他请求 | 后台处理，不阻塞查询 |
| 创建事件后可见性 | 可能不可见（快照未 patch） | 即时可见（实时查询） |
| 修一个 bug 的风险 | 高（牵一发动全身） | 低（模块隔离） |
| 单文件最大行数 | 2426 行 | ≤ 400 行 |

## 五、如果只能做一件事

如果时间极其有限，**最高优先级是砍掉快照表**。

理由：
- 快照表是阶段 52-82 中 80% 问题的直接或间接原因
- 砍掉后删除 ~2000 行代码（`snapshot_builder.py` + `alerts.py` 快照路径）
- 代码量减少、路径减少 → bug 表面积减少
- 实时查询在 SQLite 上可能慢（首次 5-10s），但**数据准确**，用户宁可等 5 秒看到正确数据，也不愿意 1 秒看到错误数据

回退后性能补偿方案：
- 保留一个简单的进程内 LRU 缓存，TTL 30 秒
- 任何写操作（事件/标签/追踪/导入）调用 `cache.clear()`，一行代码
- 不需要增量 patch、不需要版本切换、不需要覆盖层

## 六、总结

| 问题 | 根因 | 正确的解法 | 错误的解法 |
|---|---|---|---|
| 又卡又慢 | SQLite 写锁 + 全表扫描 | 换数据库或读写分离 | 加更多缓存层 |
| 标签/事件不显示 | 缓存/快照未同步 | 砍掉快照，实时查询 | 给每个变更点加 patch |
| 改一处坏一片 | 2426 行的 God File | 按职责拆分文件 | 在同一文件里加 if/else |
| 筛选反复修 | 前后端职责不清 | 所有筛选走后端 SQL | 前端拉全量再过滤 |
| 每个版本都有新 bug | 架构复杂度超过可维护上限 | 简化架构（减代码） | 加更多补丁（加代码） |

**一句话结论：不需要换语言重写，但必须重构架构。当务之急是砍掉快照表，回归简单的实时查询路径。**
