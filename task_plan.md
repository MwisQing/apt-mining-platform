# 研判工作台事件显示与创建事件修复计划

## Goal
解决“创建事件后研判工作台不显示事件状态”和“点击创建事件卡死”的问题，并收敛快照表带来的可用性风险：工作台候选查询可以继续利用快照加速，但事件/标签/追踪这类人工研判状态必须以实时业务表为准，避免快照陈旧导致平台不好用。

## Phases

| Phase | Status | Work |
| --- | --- | --- |
| 1 | complete | 梳理当前候选接口、快照响应、事件匹配和前端展示链路 |
| 2 | complete | 修改后端：快照候选补实时事件/标签/追踪状态，或在风险过高时回退实时查询 |
| 3 | complete | 补齐回归测试，覆盖创建事件后工作台立即显示 |
| 4 | complete | 前端构建/后端测试验证 |
| 5 | complete | 更新 CLAUDE.md 进度记录 |
| 6 | complete | 恢复工作台创建事件操作列和弹窗，避免点击/提交卡死 |
| 7 | complete | 收敛事件创建后端耗时路径，避免同步快照 patch 阻塞 |
| 8 | complete | 回归验证并更新 CLAUDE.md |

## Decisions

- 事件显示属于人工研判结果，不应完全依赖预计算快照的陈旧 JSON。
- 候选性能优化可以保留，但实时业务状态必须在响应前二次合成。
- 已选择“快照底表 + 实时关系覆盖”的折中方案，避免继续依赖每个写接口都完美 patch 快照。
