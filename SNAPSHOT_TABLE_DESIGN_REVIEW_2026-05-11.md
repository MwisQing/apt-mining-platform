## 1. 总结结论

这份设计文档抓对了核心问题，也选对了大方向：

- 研判工作台当前性能瓶颈不在 FastAPI 本身
- 主瓶颈确实是请求期对 9 万多条候选结果做 Python 全量装饰
- 用“候选快照表 / 物化结果表”替代“请求时全量计算”是正确路线

但是，**当前文档还不能作为可直接开发的实施稿**。原因不是思路错，而是关键实现细节存在一致性、查询语义、前后端契约和并发模型风险。

因此本次评审结论为：

- **方向性方案：认可**
- **当前实施稿：不认可直接照文档开发**

## 2. 认可的部分

### 2.1 根因判断正确

文档正确识别了当前主瓶颈：

- SQL 去重与聚合耗时仅数秒
- 主要耗时集中在 `_make_items` 与 `_decorate_candidate_items`
- 当前架构的本质问题是“请求时全量计算”

这一判断与现有代码实测相符。

### 2.2 快照化方向正确

文档提出：

- 导入后预计算候选结果
- 请求期只做 `WHERE + ORDER BY + LIMIT`

这与工作台性能目标一致，也是比继续扩展 `_full_cache` 更合理的长期方案。

### 2.3 快照字段设计整体覆盖较全

文档给出的快照表字段，基本覆盖了工作台当前返回结构所需的主要信息：

- 原始字段
- 聚合字段
- 评分字段
- 关系字段
- 热度字段
- 摘要字段
- 排序辅助字段

这说明作者已经比较深入地理解了当前 `/api/alert-candidates` 的返回结构。

## 3. 不认可直接开发的原因

以下问题是本次评审中最关键的否决项。

### 3.1 `badges_json LIKE` 不是可靠设计

文档建议将 badge 过滤改为对 `badges_json` 做 SQL `LIKE` 过滤。

这个做法不可靠，原因包括：

- `LIKE` 只能做字符串子串匹配，不是结构化精确匹配
- badge `name`、`label`、颜色等字段都可能出现在 JSON 中，容易误命中
- 一旦 badge 名称、标签文案、JSON 格式变化，查询语义会漂移
- 后续维护成本高，难以证明过滤结果是严格正确的

当前系统中 badge 逻辑本质上是结构化关系，不应降级成字符串模糊检索。

**评审建议：**

- 不要用 `badges_json LIKE`
- 优先方案：新增 `alert_candidate_snapshot_badges` 子表
- 次优方案：维护一个规范化的 badge key 列，专门用于精确过滤

### 3.2 `DELETE + INSERT OR REPLACE` 的全量重建方式不可接受

文档中的全量重建流程是：

1. `DELETE FROM alert_candidate_snapshots`
2. 批量 `INSERT OR REPLACE`

我不认可这个实现方式，原因如下：

- 重建过程中正式表会短暂为空
- 请求如果正好打进来，会看到空结果或半成品结果
- `REPLACE` 在 SQLite 中本质是 `DELETE + INSERT`
- 这会引入额外副作用，不适合做稳定快照切换

这类快照重建必须考虑读写并发下的一致性。

**评审建议：**

- 使用 `staging` 表重建
- 构建完成后做原子切换
- 或者引入 `snapshot_version`，通过版本号切换当前可读快照

换句话说：

**正式快照表不能在重建中暴露给读请求。**

### 3.3 “前端不需要任何修改”这个结论不成立

文档写明“前端不需要任何修改”，这一点我不认可。

原因是文档后面又提出：

- `filter_options.device_tags` 可留空
- `filter_options.badges` 可留空
- 设备标签和 badge 选项可由前端从当前页提取

但当前工作台前端并不是这么实现的。

现有前端行为是：

- 直接依赖后端返回 `filter_options`
- 表头筛选的备选值由后端接口提供

如果后端改成不给完整选项，现有功能会退化或行为变化。

**评审建议：**

- 前端契约必须重新明确
- 如果要保持“前端零改动”，后端必须继续提供完整且可用的 `filter_options`
- 如果不提供，文档就必须明确这是“需要前端配合修改”的方案

### 3.4 “快照构建中返回空结果”没有真正闭环到用户体验

文档考虑了“快照表为空时返回构建中状态”，这是好事，但还不完整。

问题在于：

- 当前前端没有消费 `snapshot_status`
- 当前前端收到空 `items` 和 `total=0` 时，更像“没有数据”
- 用户未必能区分“快照正在构建”和“查询结果为空”

因此这不是单纯后端返回一个 `meta.message` 就结束的事。

**评审建议：**

- 文档必须补充前端配套方案
- 至少要说明：前端如何识别“构建中”、如何展示、是否自动轮询

### 3.5 增量刷新设计低估了并发与一致性问题

文档已经提出导入、标签、追踪、事件变更都要触发快照刷新，这是对的。

但问题是，当前文档没有定义以下关键机制：

- 同一时间多个刷新任务是否允许并发执行
- 同一设备/同一 target 的多个刷新如何合并
- 全量重建与增量刷新同时发生时如何协调
- 失败后如何恢复
- 前台读请求如何避免读到旧版本与新版本混杂的数据

没有这些机制，后续大概率会出现：

- 偶发脏数据
- 结果回退
- 某些候选行刷新了，某些没刷新
- 重建完成后又被旧增量任务覆盖

**评审建议：**

- 文档必须补上刷新一致性模型
- 至少定义：
  - 单飞机制
  - 队列策略
  - 版本号或构建批次号
  - 失败恢复策略

## 4. 必须修改后才能开发的事项

以下事项是我认为的“准入条件”。不补齐，不建议进入正式开发。

### 4.1 重建机制必须改为“staging + 原子切换”

必须替换：

- `DELETE FROM 正式表`
- `INSERT OR REPLACE INTO 正式表`

建议改成：

1. 创建或清空 `alert_candidate_snapshots_build`
2. 全量写入构建表
3. 构建完成并校验通过
4. 原子切换当前版本

可选实现：

- 双表切换
- 单表加 `snapshot_version`

### 4.2 badge / device tag 过滤必须改成结构化方案

必须替换：

- 对 JSON 文本做 `LIKE`

建议方案：

- `alert_candidate_snapshot_badges(snapshot_id, badge_name, badge_label, ...)`
- `alert_candidate_snapshot_tags(snapshot_id, tag_id, tag_name, ...)`

这样：

- 过滤语义清晰
- 索引可做
- 精确匹配可靠

### 4.3 明确前端契约

必须明确：

- `filter_options` 是否仍由后端完整提供
- 构建中状态是否需要前端显示
- 若显示，交互是什么
- 首次部署时用户看到什么

不能同时写：

- “前端不需要改”
- “某些 filter_options 留空，让前端从当前页提取”

这两句话是冲突的。

### 4.4 补齐刷新一致性设计

必须在文档中明确：

- 全量重建如何与增量刷新共存
- 是否有构建版本号
- 是否有任务串行队列
- 失败时是否回退到旧版本快照
- 刷新完成标志是什么

## 5. 建议修订方向

如果继续沿这条路线推进，我建议把文档修成下面这种结构。

### 5.1 方案层次拆分

把设计分成两层：

- 第一层：目标架构
- 第二层：最小可落地版本

避免一次文档里把“理想最终态”和“简化实现”混在一起。

### 5.2 把快照表拆成主表 + 关系子表

建议不要把所有结构化关系都堆进 JSON 主表。

建议至少拆出：

- 主表：基础字段、热度、评分、摘要、状态
- badge 子表
- tag 子表

event / trace 可以先保留 JSON，但 badges/tags 最好结构化。

### 5.3 增加“版本模型”章节

必须新增章节明确：

- `active_snapshot_version`
- `building_snapshot_version`
- 查询只读 `active`
- 构建完成后切换版本

这是整份方案能否稳定落地的关键。

### 5.4 增加“前端联动”章节

建议新增一节，明确：

- 构建中 UI
- filter_options 来源
- 空结果与构建中的区分方式
- 兼容策略
